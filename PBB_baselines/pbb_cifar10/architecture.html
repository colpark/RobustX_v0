PBB-style JEPA on CIFAR-10 (40% sparse pool) i-JEPA-faithful self-supervised pretraining for sparse-input vision

A PointBigBird-style Joint Embedding Predictive Architecture trained on a 40% random-pixel sparse pool of CIFAR-10. Per-pixel BigBird sparse-attention encoder, anchor-KNN multi-block masking, direct per-pixel target gather, smooth-L1 loss in latent space with DINO-style EMA target centering before per-token LayerNorm. No VICReg / contrastive / centering-sharpening softmax: target centering + EMA + per-token LN is sufficient at this content density (24 bits of RGB per token).

1. Architecture

Two key facts about this architecture:

  • No learned compression / "pool" / "readout" layer. Targets and context features are direct gathers from the per-token encoder output. The encoder's job is to produce contextualized per-token features; the JEPA loss operates on those features directly.
  • Position-symmetric predictor. Both context tokens (in the predictor's keys) and target queries receive the same positional embedding γ_Fourier(coord), so the predictor's attention has consistent spatial grounding on both sides.

4. Novel contributions vs. i-JEPA

The core JEPA loss + EMA + multi-block masking is unchanged. What's specifically new here:

Novel 1. Multi-curve space-filling-curve permutations per encoder layer

Standard BigBird operates on a fixed sequence order; in 2D/3D we sort tokens by Hilbert or Z-order before attention. Our encoder picks a different SFC permutation per layer (uniformly random from {z, z-rev, hilbert, hilbert-rev}). Each layer's block-sparse attention covers a different local-neighborhood pattern, and stacked depth produces a globally-connected receptive field at BigBird's per-layer cost. This is essential when running BigBird on sparse 2D/3D inputs, where there is no single "canonical" sequence order.

Novel 2. Anchor-KNN-disjoint multi-block masking on sparse pools

i-JEPA's rectangular target blocks assume a regular grid. On a 40% sparse pool there is no grid, so block extent must be defined by distance. Our masking picks 4 anchors uniformly in the pool and grows each block by k-nearest pool members in 2D coordinates, with successive blocks excluding previously-claimed pixels. Context comes from one anchor in the remaining pool. This produces locally-coherent target regions that don't overlap, which is the property i-JEPA's rectangular blocks have.

Novel 3. DINO-style EMA target centering before per-token LayerNorm

This is the load-bearing anti-collapse mechanism. Per-token LayerNorm alone (the i-JEPA recipe) allows the "all targets point in the same direction across the batch" failure mode on sparse inputs. Subtracting an EMA of the per-feature batch mean before LN removes that minimum without adding any regularizer to the loss. Specifically:

target_center.update(h_tgt_raw)                                # EMA: c ← 0.9·c + 0.1·mean(h_tgt_raw)
h_tgt = F.layer_norm(target_center(h_tgt_raw), (D,))          # subtract c, then LN per token

This addition is 5 lines and replaces what VICReg / centering+sharpening softmax / variance-covariance regularizers do in heavier recipes.

Novel 4. pos_symmetric=True predictor

Both the predictor's context tokens (keys/values) and its mask-token queries receive an explicit γ(coord) positional embedding before the dense Transformer. Standard i-JEPA only adds position to queries. Symmetric position avoids a positional-information asymmetry between context and target sides that would otherwise force the predictor to bridge two different coordinate frames.

Borrowed from i-JEPA / DINO / BigBird

  • JEPA loss formulation (smooth-L1 in latent space, EMA target, stop-grad).
  • Multi-block masking discipline (disjoint context and target sets).
  • BigBird block-sparse attention (window + random + global blocks).
  • DINO target centering operator.
  • Gaussian Fourier feature positional encoding.

5. Why this works (the theory of operation)

The JEPA loss has several low-loss minima:

  1. Useful representation minimum. Encoder produces content-aware per-pixel features; predictor learns to extrapolate spatially; loss is low because predictions actually match content.
  2. Constant-output minimum. Encoder maps every input to the same direction; predictor outputs that direction; per-token LN makes them all unit-norm; loss is zero. Probe is at chance.

What pushes gradient descent toward (1) rather than (2):

  • High per-token content (24 bits of RGB per pixel). Random encoder weights applied to genuinely different RGB pixels already produce different output vectors at initialization. The constant minimum is far from init in weight space.
  • DINO-style centering before LN. If the encoder collapsed to constant direction v, the EMA batch mean would equal v, centering would zero it out, and per-token LN would amplify residual noise into different per-token directions across the batch. The constant minimum is no longer at zero loss.
  • 200 target positions per sample. Dense supervision: every batch produces gradients at 200 specific spatial locations, forcing the encoder to produce position-discriminating features.
  • EMA stop-grad. Standard JEPA stability mechanism — the target is a slowly-moving target the online encoder chases.

7. Caveats and honest limitations

Linear probe on CIFAR-10 with sparse pretraining has a real ceiling. The pretraining task only sees 40% of pixels and the test path sees 20%. The probe is fundamentally evaluating "can a partial view of a 32×32 image classify into 10 classes". Published numbers for SSL methods on CIFAR-10 with no augmentation (the cleanest comparison) typically land in the 30–60% linear-probe range. With strong augmentation and longer pretraining, this can push past 70%.
The recipe is i-JEPA + sparse-input adaptations + target centering — not a fundamentally new SSL idea. The novel content is the integration: BigBird sparse encoder, KNN multi-block masking, target centering for sparse inputs. These are reasonable engineering choices for the sparse-input regime, not new SSL principles.
