PBB-style JEPA on CIFAR10-DVS i-JEPA-faithful self-supervised pretraining on per-event tokens

A direct port of the CIFAR-10 PBB recipe to event-camera data. Per-event BigBird sparse-attention encoder over 8192 (x, y, t, polarity) tokens, anchor-KNN multi-block masking in 3D coordinates, direct per-event target gather, smooth-L1 loss in latent space with DINO-style EMA target centering before per-token LayerNorm. The same architectural recipe as the CIFAR-10 variant; the only changes are coord_dim, signal_dim, and pool/block sizes.

Modality caveat. Each event token carries 1 bit of polarity (vs. 24 bits of RGB per pixel in the CIFAR-10 variant). This is closer to the per-token entropy limit where naked JEPA reliably works without further architectural defenses. Provided here as the cleanest possible event baseline for analyzing JEPA dynamics at low per-token content.

3. Why this might underperform the CIFAR-10 variant

The JEPA loss has two attractors:

  1. Useful representation — encoder learns content-aware per-event features; predictor learns extrapolation; loss low because predictions match content.
  2. Constant-output — encoder maps every event to the same direction; per-token LN normalizes; predictor outputs the constant; loss zero; probe at chance.

What pushes gradient descent toward (1) is per-token content density. At random initialization, the encoder applied to genuinely different inputs produces genuinely different outputs only if the inputs themselves differ at the token level. With 1-bit polarity per event, two adjacent events with the same polarity are almost indistinguishable token-wise — the only thing differentiating them is the coordinate, which the encoder treats as a soft positional signal.

DINO-style centering pushes the constant-direction minimum away by forcing target features to vary across the batch dimension. This is enough at CIFAR-10's 24-bit content level — the encoder's initialization-time per-token diversity is already large enough that the centering term holds. At 1-bit content the margin is thinner; whether centering alone holds is the empirical question this notebook addresses.

4. Why use this folder anyway

This is the cleanest possible PBB baseline on event data — same recipe as the working CIFAR-10 variant, no additional architectural defenses, no regularizers. It serves three purposes:

  • Diagnostic. If naked PBB collapses on events but works on CIFAR-10, that pinpoints content density (not the recipe) as the load-bearing variable. This is the cleanest test of the modality-sensitivity hypothesis.
  • Reference point. Any heavier event-SSL recipe (cross-attention readout, voxelization, temporal causality, VICReg) can be evaluated against this baseline to quantify what the addition actually buys.
  • Reproducibility. Same library, same architectural code as the CIFAR-10 variant — apples-to-apples comparison across modalities.

7. Caveats

Per-event JEPA on raw 1-bit-content tokens is an active research question, not a solved benchmark. No published work runs this exact recipe on raw events; all published event-SSL either voxelizes (collapses time into bins) or aggregates events into multi-event patch tokens. The naked PBB recipe applied here is closer to a "what does JEPA look like at the lowest per-token entropy?" sanity check than to a competitive baseline.
