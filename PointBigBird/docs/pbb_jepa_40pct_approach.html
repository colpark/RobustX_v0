PointBigBird-JEPA Sparse-input self-supervised pretraining on CIFAR-10 — the 40 % approach

1. The problem: sparse-input self-supervised learning

Each CIFAR-10 image has 32 × 32 = 1024 pixels. We impose a hard constraint: only a fixed 40 % subset (410 pixels) is ever available per image — at any phase of training or testing. The model never sees the other 60 % of any image.

This is a stand-in for sparse-input regimes that arise in scientific data (point clouds, masked images, sensor networks, sparse-view reconstruction). We want to know whether self-supervised pretraining can produce features that classify well under this constraint — and we want an architecture that handles sparse, irregular inputs natively rather than being shoe-horned from a dense-grid design.

Why this is hard: humans struggle to recognize CIFAR-10 classes at 20 % pixel density. Information-theoretically, a lot of detail is gone. A "good" linear-probe result on 20-40 % sparse views sits in the 65–75 % band, with an estimated ceiling around 85 % (what a strong supervised model trained on 40 % sparse views could achieve).

2. Data budget and multi-block masking

The data pipeline has two tiers of randomness.

Tier 1 — the 40 % pool, fixed per image

# Determined once per image with a global seed (pool_seed = 12345)
rng = np.random.RandomState(pool_seed)
pool_idx = rng.permutation(1024)[:410]   # 410 pixel indices, identical at every epoch

Image #7's pool is exactly the same 410 pixels at every iteration of training, every probe call, every test pass. This is the "data budget" — that's all there is.

Tier 2 — context and target blocks, resampled each training step

We follow i-JEPA's MaskCollator recipe (multi-block masking with disjoint targets), adapted for the pool constraint:

  1. Sample N_PRED = 4 target anchors uniformly from the 410-pool; each anchor's K_TGT = 50 nearest pool pixels form a contiguous target block. With disjoint_targets=True, anchors avoid positions already in earlier target blocks.
  2. Build forbidden = ⋃ target_blocks.
  3. Sample one context anchor from pool \ forbidden; take its K_CTX = 100 nearest neighbors within the allowed region as the context block.
full image (1024 px)
  └── 40% pool (410 px, fixed per image)         ← all training data
        ├── 4 target blocks (200 positions total, contiguous, disjoint from each other)
        ├── 1 context block (100 px, contiguous, disjoint from targets)
        └── ~110 slack pool pixels (unused this iteration)

Disjointness context ∩ targets = ∅ is invariant by construction (this was the v8 fix that broke the 0.85 plateau in earlier OmniField-JEPA runs: previously target positions could fall inside the context, making the predictor's job a retrieval task instead of a prediction task).

Test-time contract

At test/probe time we serve the full K_POOL = 410 pool (all 40 %) deterministically. Earlier versions served only the first K_HALF = 205 ("legacy 20 %" contract); the v2 setup defaults to k_test = 410 via cfg.test_full_pool = True.

3. Tokenization

Each visible point becomes a single D_model = 256-dim token:

tokenn = Wsignal · rgbn  +  Wpos · γ(coordn)

where γ(c) = [sin(B·c), cos(B·c)] are Gaussian Fourier features with B ∈ ℝ²×⁹⁶ fixed-random (scale = 15). Both the signal projection W_signal and the positional projection W_pos map into the same D_model-dim space, so token = content + position, added.

Why positional embedding is critical here: the architecture re-orders tokens along different curves at every layer (see §4), so sequence position is uninformative — it means different things at each layer. The model can only identify "where am I in space?" by reading γ(coord). Position information is decoupled from sequence order by design.

4. Encoder: space-filling curves + BigBird block-sparse attention

The encoder is a stack of L = 6 Transformer-style blocks. Each block has three non-standard moves on top of the usual (LayerNorm → self-attention → FFN):

4a. Per-layer random serialization

For each block, we sample one ordering uniformly from {z-order, z-order-reversed, Hilbert, Hilbert-reversed} and reorder the token sequence accordingly. After the attention + FFN, we reorder back.

Curves are global functions of the 32 × 32 grid; they are precomputed once at module load. For a specific sample with K visible pool pixels, the per-sample sequence is just argsort(global_rank[visible_pixel_ids]) — O(K log K). No per-sample curve re-walking, no learned parameters. 

# pbb/serialization.py
def subset_perm(subset_pixel_ids, full_rank):
    ranks = full_rank[subset_pixel_ids]    # look up each visible pixel's curve rank
    return torch.argsort(ranks)            # sort the subset by rank
Locality: Hilbert is locality-preserving (consecutive curve positions are L1-adjacent in image space), so adjacent visible pixels along the curve are typically spatial neighbors. Z-order is weaker but has different locality structure. Reverses provide two more distinct sequence neighborhoods. Across layers, every pixel gets to "see" its spatial neighborhood under multiple 1-D slices.

4b. BigBird block-sparse self-attention

Within the chosen ordering, the sequence is split into blocks of block_size = 8. Each query block attends to:

  • (2·window + 1) = 3 window blocks (self + left + right).
  • n_global = 2 "global" blocks (first and last in the sequence).
  • n_random = 2 blocks sampled uniformly per forward pass.

Total: 7 attended blocks per query block, regardless of sequence length.

SequenceN blocksSparsity (attended / total)
Train context (K_CTX = 100, padded to 104)137/13 ≈ 54 %
Probe / test (K_TEST = 410, padded to 416)527/52 ≈ 13 %
Target encoder (K_POOL = 410, padded to 416)527/52 ≈ 13 %

Implementation is pure PyTorch via index_select/gather on the block dimension. No custom CUDA kernels. The implementation is verified to be equivalent to dense multi-head attention when configured for full-pattern coverage (BigBirdSparseAttention(equivalent_to_dense=True) matches plain MHA to max|Δ| < 1e-5).

4c. Positional re-injection (off by default in v2)

A subtle but important point: each block can optionally re-inject the positional embedding as a residual right before attention: 

def forward(self, x, perm, inverse_perm, pos_emb, ...):
    if pos_emb is not None:        # cfg.reinject_pos
        x = x + pos_emb            # ensure every layer has access to true (y, x)
    x_p = gather(x, perm)
    x_p = x_p + attn(LN(x_p))
    x_p = x_p + ffn(LN(x_p))
    return gather(x_p, inverse_perm)

Without re-injection, the input-stage positional information dilutes through deep layers, and the per-layer reshuffling can't motivate the model to attend to true position (sequence position is meaningless under randomization). The v2 + input-pos-only variant turns re-injection OFF and accepts the trade-off; an ablation toggle is provided. 

┌── 6 × EncoderBlock ────────────────────────────────────────────────┐
│                                                                    │
│   for each layer ℓ ∈ 1..6:                                          │
│     pick ordering π_ℓ ∈ {z, z_rev, hilbert, hilbert_rev}            │
│     x' = gather(x, π_ℓ)                                             │
│     [optional: x' = x' + pos_emb]                                   │
│     x' = x' + BigBirdSparseAttn(LayerNorm(x'))                      │
│     x' = x' + FFN(LayerNorm(x'))                                    │
│     x  = gather(x', π_ℓ⁻¹)        ← back to canonical order          │
│                                                                    │
└─→ g (B, K, D_model)                                                 │

5. Predictor

A small 4-layer dense Transformer (no BigBird, no serialization) that takes the context encoder's per-token features and predicts the target encoder's features at the 200 target coordinates. 

# pbb/model.py PBBPredictor.forward
ctx_tok = proj_g(g_ctx)                              # (B, K_ctx, D_pred)
if pos_symmetric:                                     # v2 #2
    ctx_tok = ctx_tok + proj_pos(γ(ctx_coords))
tgt_tok = proj_pos(γ(target_coords)) + mask_token    # (B, N_tgt, D_pred)
x = concat([ctx_tok, tgt_tok])
for blk in 4 × PredictorBlock:                        # dense MHA + FFN
    x = blk(x)
h_pred = proj_out(LayerNorm(x[K_ctx:]))               # take target slice → D_model
Positional symmetry (v2 fix #2): the original i-JEPA gives positional information to both the context and target tokens of the predictor. Earlier versions of this code only added γ(coord) to the target tokens, forcing the predictor to infer "where in space" each g_ctx token came from purely from its content. The v2 fix passes ctx_coords through and adds γ(ctx_coords) to ctx_tok too. Shares proj_pos with the target side, no new parameters.

Target features — direct gather, not soft-pool

By construction, target_coords ⊂ pool_coords, so the target encoder's per-token features directly contain the features at each target position. We gather them: 

# pbb/jepa.py
h_tgt_raw = gather(g_tgt, tgt_pool_pos)              # (B, N_tgt, D_model)
h_tgt     = LayerNorm(h_tgt_raw)                      # cfg.use_centering=False (canonical i-JEPA)

No learnable parameters on the target side. Earlier versions of this code (OmniField-JEPA v1–v8) used a deterministic Gaussian soft-pool because the latent set didn't correspond 1-to-1 with input points; with the per-token PointBigBird architecture, direct gather is correct and simpler.

6. Loss, EMA target, and centering

Loss

Two options, toggled by cfg.loss_type:

loss = F.smooth_l1_loss(h_pred, h_tgt)                          # "smooth_l1" (legacy)
loss = (1 - F.cosine_similarity(h_pred, h_tgt, dim=-1)).mean()  # "cosine" (v2 default)

Cosine is direction-only — it doesn't pull ‖h_pred‖ toward ‖h_tgt‖. Over long training, smooth-L1's magnitude pressure tends to eat capacity that should go to discriminative directions, so v2 switches to cosine.

EMA target encoder

m = ema_start (0.999) → ema_end (0.9999) linearly over total steps
for p_q, p_k in zip(context_encoder.parameters(), target_encoder.parameters()):
    p_k.data.mul_(m).add_((1 - m) * p_q.detach())

The target encoder is the EMA-smoothed shadow of the context encoder, frozen w.r.t. gradients. v2 caps ema_end at 0.9999 instead of 1.000 so the target keeps moving forever — avoids the "predictor saturation" failure mode where features drift after the target encoder freezes.

Centering — dropped in v2

DINO-style centering (running EMA of the per-feature mean, subtracted from the target) is not part of the canonical i-JEPA recipe. It was carried over from v1–v8 of the OmniField-JEPA line. Dropped in v2: 

# cfg.use_centering = False  (new default)
h_tgt = F.layer_norm(h_tgt_raw, (h_tgt_raw.size(-1),))     # no center subtraction

With centering off, we rely on the canonical i-JEPA anti-collapse stack: EMA target encoder + asymmetric masking + multi-block targets + LayerNorm of h_tgt.

8. Anti-degradation mitigations

JEPA's downstream probe accuracy often peaks ~200 epochs in and then drifts down ("predictor saturation + spectral collapse + smooth-L1's magnitude pull"). v2 adds four mitigations:

#MitigationToggle
M1Save best-probe-acc checkpoint, not just best-lossalways on; pbb_probe_best.pt
M2Early-stop when probe acc hasn't improved for N consecutive checkscfg.probe_patience = 3
M3Cap EMA momentum below 1.000 so target encoder never freezescfg.ema_end = 0.9999
M5Cosine similarity loss instead of smooth-L1 (no magnitude pull)cfg.loss_type = "cosine"
M6Drop DINO-style centering (not part of canonical i-JEPA)cfg.use_centering = False
Why loss can rise while probe accuracy rises: the smooth-L1 / cosine loss against a moving EMA target is a proxy. Once the encoder produces more discriminative features, the target distribution shifts faster than the predictor can chase, inflating the proxy loss even as features genuinely improve. This is a well-known JEPA pitfall — never use loss-min for the final checkpoint. Trust the probe.

9. Evaluation: linear probing

The probe protocol is the standard SSL convention with two extensions for our setup:

  1. Encoder is frozen (requires_grad_(False) saved + restored).
  2. Head is either a single Linear(D_model, 10) (legacy) or an AttnPoolHead + Linear pair (v2 default). The AttnPool head adds a single learnable query that cross-attends to the encoder's per-token features, producing one D-dim vector per image.
  3. Probe is trained on the CIFAR-10 train set (50 K images) using the test-mode sampler — same fixed deterministic 40 % view that the test loader uses. So probe-train and probe-eval see the same input distribution.

Expected band: on this sparse-input setup, a "good" probe result is 50–65 %, "nice / publication-worthy" is 65–75 %, approaching the supervised ceiling of ~85 % at 40 % pixel density. The v2 setup typically lands in the upper end of the "good" range and climbs into "nice" with longer training (200+ epochs), provided you keep the probe-best checkpoint rather than the loss-best one.
